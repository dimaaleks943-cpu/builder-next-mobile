### Общие принципы `builder`

Этот файл фиксирует ключевые идеи и договорённости по компонентам конструктора
(`builder/src/craft`) и их связи с рентаймом (`site-runtime-ssr`) и будущей мобилкой.
Идея такая же, как у `site-runtime-ssr/DEV_NOTES.md`: быстро вспомнить архитектуру
и продолжать работу в том же стиле.

---

### 1. Базовые компоненты Craft: Body, Block, Text, LinkText

#### `Body`

- Файл: `src/craft/Body.tsx`.
- Роль:
  - Корневой контейнер страницы в билдере.
  - Определяет общую область, куда можно добавлять другие компоненты (`Block`, `ContentList` и т.п.).
- В рентайме:
  - Есть одноимённый компонент `Body` в `site-runtime-ssr/components/Body.tsx`.
  - При сериализации через Craft JSON узел `Body` превращается в `ComponentNode` с `type: "Body"`,
    который `renderer` рендерит через `componentMap["Body"]`.
- Для мобилки:
  - Логика та же: корневой layout‑контейнер; реализация UI будет другой, но структура дерева и имя типа (`"Body"`) сохраняются.

#### `Block`

- Файл: `src/craft/Block.tsx`.
- Роль:
  - Универсальный контейнер с отступами/фоном/выравниванием.
  - Используется как строительный блок для раскладки внутри `Body`, `ContentList` и т.п.
- В рентайме:
  - Аналогичный компонент `Block` в `site-runtime-ssr/components/Block.tsx`.
  - В JSON Craft узел блока → `ComponentNode` `type: "Block"` с пропсами стилизации.
- Для мобилки:
  - Сохраняем те же пропсы по смыслу (margin/padding/выравнивание и т.п.),
    но маппим их на соответствующие стили/виджеты мобилки.

#### `Text`

- Файл: `src/craft/Text.tsx`.
- Роль:
  - Текстовый компонент, который редактируется в билдере.
  - Умеет работать в двух режимах:
    - **Manual** — статический текст, который задаёт пользователь.
    - **Collection / data‑binding** — значение берётся из поля коллекции.
- Дополнительные особенности:
  - Модалка настроек (`RightPanel`) позволяет:
    - переключать режим Manual / Collection,
    - выбирать поле коллекции (`collectionField`) при работе внутри `ContentList`.
- В рентайме:
  - `site-runtime-ssr/components/Text.tsx` повторяет ключевые пропсы (включая `collectionField`),
    но **не знает** про Craft, только про данные и стили.
  - Для динамических данных использует хук `useContentData()` (см. `ContentDataContext` в рентайме).
- Для мобилки:
  - Тот же контракт пропсов (`text`, `collectionField`, стили),
    реализация — нативный текстовый виджет.

#### `LinkText`

- Файл: `src/craft/LinkText.tsx`.
- Роль:
  - Текстовый компонент со ссылкой (URL, поведение при клике).
- В рентайме:
  - Аналогичный компонент `LinkText` в `site-runtime-ssr/components/LinkText.tsx`,
    который рендерит `<a>` или другой ссылочный элемент.
- Для мобилки:
  - Аналогичный по пропсам компонент, но с навигацией/линками мобильного UI.

### 2. ContentList и работа с коллекциями в билдере

#### Общая идея

`ContentList` в билдере — это **редактируемый список по коллекции**:

- Пользователь выбирает источник данных (коллекцию).
- Настраивает шаблон ячейки (составляет его из `Text`, `Block`, `LinkText` и др.).
- Билдер сам размножает этот шаблон по всем элементам коллекции.

В рентайме (`site-runtime-ssr`) аналогичный компонент `ContentList`:

- получает коллекцию по `selectedSource` из API,
- рендерит шаблон ячейки для каждого элемента,
- подставляет данные элемента через общий контекст `ContentDataContext`.

#### `ContentList`

- Файл: `src/craft/ContentList.tsx`.
- Роль в билдере:
  - Основной визуальный компонент списка.
  - Позволяет:
    - выбрать коллекцию (`selectedSource`),
    - задать `itemsPerRow` (кол-во элементов в строке),
    - редактировать содержимое первой ячейки как шаблон.
- Внутренняя механика:
  - Для ячеек используются специальные элементы с id вроде `contentListId-cell-0`, `contentListId-cell-1` и т.п.
  - Узлы ячеек могут храниться как в `data.nodes`, так и в `data.linkedNodes`.
  - Есть вспомогательные функции для:
    - синхронизации шаблона между ячейками,
    - клонирования дерева узлов при добавлении/удалении ячеек.
- Сериализация:
  - При сохранении страницы дерево Craft (включая `ContentList`) сериализуется в JSON.
  - В рентайме `craftContentToComponents` специально разбирает `ContentList`,
    вытаскивая **шаблон ячейки** из первой ячейки и превращая его в `children` узла `ContentList`.

#### `ContentListCell` и контексты

- Файлы:
  - `src/craft/ContentListCell.tsx`
  - `src/craft/ContentListCellContext.tsx`
  - `src/craft/ContentListDataContext.tsx`

- Роль:
  - `ContentListCell` — обёртка/«ячейка» внутри `ContentList` в билдере.
  - Контексты:
    - `ContentListCellContext` — информация о позиции ячейки (индекс, состояние и т.п.).
    - `ContentListDataContext` — данные коллекции, которые могут использоваться в билдере
      при настройке (например, для предпросмотра).

- В рентайме:
  - `ContentListCell` **не рендерится напрямую**: при разборе JSON мы поднимаем его детей
    в `children` у узла `ContentList`.
  - За передачу данных элемента коллекции отвечает уже общий `ContentDataContext`.

- Для мобилки:
  - Логика аналогична рентайму: `ContentListCell` используется только в билдере,
    а реальные данные и шаблон рендерятся через общий механизм data‑binding’а.

### 3. Связь builder ↔ site-runtime-ssr ↔ будущая мобилка

Ключевые договорённости:

1. **Имена компонентов и пропсов**:
   - Билдерские компоненты (`Body`, `Block`, `Text`, `LinkText`, `ContentList`) задают
     **контракт** для рентайма и мобилки:
     - типы (`type: "Text"`, `"Block"`, `"ContentList"` и т.п.),
     - структура пропсов (например, `Text` обязательно имеет `text`, может иметь `collectionField`).
   - Рентайм и мобилка реализуют визуальную часть, но **соблюдают эти контракты**.

2. **Сериализация**:
   - Вся страница — это дерево Craft, которое сериализуется в JSON.
   - Этот JSON должен быть:
     - достаточно стабильным, чтобы рентайм/мобилка могли его парсить,
     - по возможности независимым от внутренних деталей Craft (используем `resolvedName`/`displayName` и
       аккуратную работу с `nodes`/`linkedNodes`).

3. **Data binding (коллекции и динамические данные)**:
   - В билдере:
     - `ContentList` и связанные контексты дают возможность настраивать шаблон
       и видеть пример данных.
     - Компоненты типа `Text` хранят в пропсах `collectionField` и т.п., но
       не ходят в API сами.
   - В рентайме/мобилке:
     - контейнеры (список, таблица) получают данные из API/модели,
     - передают `itemData` вниз через `ContentDataContext` (или аналогичный общий контекст),
     - простые компоненты (`Text`) просто читают этот контекст и подставляют значение.

### 4. Стиль кода для `builder`

Чтобы `builder` и `site-runtime-ssr` выглядели и ощущались одинаково:

- **Функции**
  - Предпочитаем **стрелочные функции**:
    - компоненты вида `export const Text = (...) => { ... }`,
    - коллбеки (`map`, `filter`, обработчики событий).
  - Обычные `function` используем только при явной необходимости работы с `this`/контекстом.

- **Типы: `interface` vs `type`**
  - Для пропсов, контекстов и объектных структур — **`interface` по умолчанию**.
  - `type` используем для:
    - union/intersection/utility типов,
    - alias’ов, где `type` реально удобнее/короче.

- **Форматирование типов**
  - В интерфейсах и объектных `type` **после каждой строки ставим `;`**:
    - `field: string;`
    - `optional?: number;`
  - Это синхронизирует стиль с `site-runtime-ssr` и упрощает чтение/диффы.

- **React‑хуки**
  - Предпочитаем **импортировать хуки из `react` поимённо** и вызывать их напрямую:
    - `import { useState, useCallback, useEffect } from "react"`
    - `useState(...)`, `useCallback(...)` и т.п.
  - Не используем стиль `React.useState`, `React.useCallback` и т.д.,
    чтобы код был короче и единообразнее.

- **Точки с запятой**
  - Не используем «leading semicolon» (`;foo()` на новой строке).
  - Пишем по классике:
    - `const x = 1;`
    - `foo();`
  - Если нужно разорвать выражения — просто ставим `;` в конце строки,
    а следующую строку начинаем без ведущей точки с запятой.

### 5. Как дальше развивать документацию

- При добавлении новых Craft‑компонентов:
  - описывать:
    - их назначение в билдере,
    - ожидаемое поведение в рентайме и мобилке,
    - особенности сериализации (что попадает в JSON, какие пропсы критичны).
- При изменении существующих:
  - фиксировать изменения контракта (новые пропсы, изменения структуры),
  - отмечать, как это влияет на `site-runtime-ssr` и на будущую мобильную реализацию.
