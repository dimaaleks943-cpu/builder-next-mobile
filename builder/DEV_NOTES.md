### Общие принципы `builder`

Этот файл фиксирует ключевые идеи и договорённости по компонентам конструктора
(`builder/src/craft`) и их связи с рентаймом (`site-runtime-ssr`) и будущей мобилкой.
Идея такая же, как у `site-runtime-ssr/DEV_NOTES.md`: быстро вспомнить архитектуру
и продолжать работу в том же стиле.

---

### 1. Базовые компоненты Craft: Body, Block, Text, LinkText, Image

#### `Body`

- Файл: `src/craft/Body.tsx`.
- Роль:
  - Корневой контейнер страницы в билдере.
  - Определяет общую область, куда можно добавлять другие компоненты (`Block`, `ContentList` и т.п.).
- В рентайме:
  - Есть одноимённый компонент `Body` в `site-runtime-ssr/components/Body.tsx`.
  - При сериализации через Craft JSON узел `Body` превращается в `ComponentNode` с `type: "Body"`,
    который `renderer` рендерит через `componentMap["Body"]`.
- Для мобилки:
  - Логика та же: корневой layout‑контейнер; реализация UI будет другой, но структура дерева и имя типа (`"Body"`) сохраняются.

#### `Block`

- Файл: `src/craft/Block.tsx`.
- Роль:
  - Универсальный контейнер с отступами/фоном/выравниванием.
  - Используется как строительный блок для раскладки внутри `Body`, `ContentList` и т.п.
- В рентайме:
  - Аналогичный компонент `Block` в `site-runtime-ssr/components/Block.tsx`.
  - В JSON Craft узел блока → `ComponentNode` `type: "Block"` с пропсами стилизации.
- Для мобилки:
  - Сохраняем те же пропсы по смыслу (margin/padding/выравнивание и т.п.),
    но маппим их на соответствующие стили/виджеты мобилки.

#### `Text`

- Файл: `src/craft/Text.tsx`.
- Роль:
  - Текстовый компонент, который редактируется в билдере.
  - Умеет работать в двух режимах:
    - **Manual** — статический текст, который задаёт пользователь.
    - **Collection / data‑binding** — значение берётся из поля коллекции.
- Дополнительные особенности:
  - Модалка настроек (`RightPanel`) позволяет:
    - переключать режим Manual / Collection,
    - выбирать поле коллекции (`collectionField`) при работе внутри `ContentList`.
- В рентайме:
  - `site-runtime-ssr/components/Text.tsx` повторяет ключевые пропсы (включая `collectionField`),
    но **не знает** про Craft, только про данные и стили.
  - Для динамических данных использует хук `useContentData()` (см. `ContentDataContext` в рентайме).
- Для мобилки:
  - Тот же контракт пропсов (`text`, `collectionField`, стили),
    реализация — нативный текстовый виджет.

#### `LinkText`

- Файл: `src/craft/LinkText.tsx`.
- Роль:
  - Текстовый компонент со ссылкой (URL, поведение при клике).
- В рентайме:
  - Аналогичный компонент `LinkText` в `site-runtime-ssr/components/LinkText.tsx`,
    который рендерит `<a>` или другой ссылочный элемент.
- Для мобилки:
  - Аналогичный по пропсам компонент, но с навигацией/линками мобильного UI.

#### `Image`

- Файл: `src/craft/Image.tsx`.
- Роль:
  - Компонент изображения с двумя вариантами источника картинки:
    1. **Ручной URL (Manual)** — пользователь вводит ссылку на картинку (внешняя ссылка или из будущего хранилища). Хранилище картинок (загрузка пользователем и выбор из него) планируется реализовать позже.
    2. **Коллекция (Collection)** — доступно **только если** компонент размещён внутри `ContentList` (позже — и других динамических контейнеров). Пользователь выбирает поле коллекции, из которого берётся URL изображения для каждой ячейки.
- Особенности:
  - Поле коллекции для картинки может приходить с API в виде **объекта**, а не строки (например, `{ urls: { original: { url: "..." }, small: { url: "..." } }, sort: null }`). В коде реализована **временная заглушка**: из такого объекта извлекается URL по `url`, `urls.small.url` или `urls.original.url`. В будущем здесь можно добавить более гибкую обработку форматов (массивы картинок, другие структуры).
  - Настройки через инлайн‑модалку (`InlineSettingsModal`) и бирку (`InlineSettingsBadge`), как у `Text` и `ContentList`.
  - При переключении Manual ↔ Collection выбранное поле коллекции сохраняется (аналогично логике в `Text`).
- В рентайме:
  - Компонент `Image` в `site-runtime-ssr` рендерит `<img>`; для источника использует либо `src` (ручной URL), либо значение из `useContentData()` по `collectionField`, с той же логикой извлечения URL из объекта.
- Для мобилки:
  - Тот же контракт пропсов (`src`, `collectionField`, размеры, стили); отображение — через нативный компонент изображения.

### 2. ContentList и работа с коллекциями в билдере

#### Общая идея

`ContentList` в билдере — это **редактируемый список по коллекции**:

- Пользователь выбирает источник данных (коллекцию).
- Настраивает шаблон ячейки (составляет его из `Text`, `Block`, `LinkText` и др.).
- Билдер сам размножает этот шаблон по всем элементам коллекции.

В рентайме (`site-runtime-ssr`) аналогичный компонент `ContentList`:

- получает коллекцию по `selectedSource` из API,
- рендерит шаблон ячейки для каждого элемента,
- подставляет данные элемента через общий контекст `ContentDataContext`.

#### `ContentList`

- Файл: `src/craft/ContentList.tsx`.
- Роль в билдере:
  - Основной визуальный компонент списка.
  - Позволяет:
    - выбрать коллекцию (`selectedSource`),
    - задать `itemsPerRow` (кол-во элементов в строке),
    - редактировать содержимое первой ячейки как шаблон.
- Внутренняя механика:
  - Для ячеек используются специальные элементы с id вроде `contentListId-cell-0`, `contentListId-cell-1` и т.п.
  - Узлы ячеек могут храниться как в `data.nodes`, так и в `data.linkedNodes`.
  - Есть вспомогательные функции для:
    - синхронизации шаблона между ячейками,
    - клонирования дерева узлов при добавлении/удалении ячеек.
- Сериализация:
  - При сохранении страницы дерево Craft (включая `ContentList`) сериализуется в JSON.
  - В рентайме `craftContentToComponents` специально разбирает `ContentList`,
    вытаскивая **шаблон ячейки** из первой ячейки и превращая его в `children` узла `ContentList`.

#### `ContentListCell` и контексты

- Файлы:
  - `src/craft/ContentListCell.tsx`
  - `src/craft/ContentListCellContext.tsx`
  - `src/craft/ContentListDataContext.tsx`

- Роль:
  - `ContentListCell` — обёртка/«ячейка» внутри `ContentList` в билдере.
  - Контексты:
    - `ContentListCellContext` — информация о позиции ячейки (индекс, состояние и т.п.).
    - `ContentListDataContext` — данные коллекции, которые могут использоваться в билдере
      при настройке (например, для предпросмотра).

- В рентайме:
  - `ContentListCell` **не рендерится напрямую**: при разборе JSON мы поднимаем его детей
    в `children` у узла `ContentList`.
  - За передачу данных элемента коллекции отвечает уже общий `ContentDataContext`.

- Для мобилки:
  - Логика аналогична рентайму: `ContentListCell` используется только в билдере,
    а реальные данные и шаблон рендерятся через общий механизм data‑binding’а.

#### Актуальный контракт collections/content

Для `ContentList` используем единый контракт во всех рантаймах:

- `selectedSource` хранит `content_type_id` (UUID), а не условные значения вроде `products`.
- загрузка items выполняется через `GET /v3/sites/{domain}/content/items` с filter:
  `{"content_type_id":["<UUID>"]}`.
- `GET /v3/sites/{domain}/content/types` - опциональный endpoint для списка типов контента
  (UI выбора в билдере), но не обязательный для runtime-рендера.
- сигнатура адаптера должна оставаться в виде `getCollectionByKey(domain, key)`, где
  `key` в текущем контракте равен `content_type_id`.

Важно: в документации и комментариях для источника `ContentList` не используем legacy wording
`products/ecommerce`; корректный термин - `content_type_id` / тип контента.

### 3. Связь builder ↔ site-runtime-ssr ↔ мобилка

Ключевые договорённости:

1. **Имена компонентов и пропсов**:
   - Билдерские компоненты (`Body`, `Block`, `Text`, `LinkText`, `ContentList`, `Image`) задают
     **контракт** для рентайма и мобилки:
     - типы (`type: "Text"`, `"Block"`, `"ContentList"` и т.п.),
     - структура пропсов (например, `Text` обязательно имеет `text`, может иметь `collectionField`).
   - Рентайм и мобилка реализуют визуальную часть, но **соблюдают эти контракты**.

2. **Сериализация**:
   - Вся страница — это дерево Craft, которое сериализуется в JSON.
   - Этот JSON должен быть:
     - достаточно стабильным, чтобы рентайм/мобилка могли его парсить,
     - по возможности независимым от внутренних деталей Craft (используем `resolvedName`/`displayName` и
       аккуратную работу с `nodes`/`linkedNodes`).

3. **Data binding (коллекции и динамические данные)**:
   - В билдере:
     - `ContentList` и связанные контексты дают возможность настраивать шаблон
       и видеть пример данных.
     - Компоненты типа `Text` хранят в пропсах `collectionField` и т.п., но
       не ходят в API сами.
   - В рентайме/мобилке:
     - контейнеры (список, таблица) получают данные из API/модели,
     - передают `itemData` вниз через `ContentDataContext` (или аналогичный общий контекст),
     - простые компоненты (`Text`) просто читают этот контекст и подставляют значение.

4. **Разделение конструктора: режимы Web и RN**  
   В билдере один интерфейс, но **два режима** конструирования и **два хранилища** контента:

   - **Режим переключается** в хедере сегментами «Веб» / «Мобилка» (`BuilderModeContext`, `MODE_TYPE.WEB` | `MODE_TYPE.RN`).
   - **Хранение на бэкенде:**
     - `content` — JSON дерева Craft для **веб‑рендера** (site-runtime-ssr).
     - `content_mobile` — JSON дерева Craft для **мобильного нативного рендера** (React Native).
   - **В билдере:** при загрузке страницы подтягиваются оба поля; при переключении режима холст заполняется соответствующим контентом (`initialContent` из `contentWeb` или `contentMobile`). Если мобильный контент пустой — используется пустое дерево Craft (ROOT + Body без детей), чтобы холст очищался, а не показывал веб‑контент.
   - **При сохранении** в API уходят оба поля: для текущего режима — сериализованное состояние редактора, для другого — последнее сохранённое значение из контекста.
   - **Общий пул компонентов**, но ограничения и дефолты зависят от режима:
     - **Block:** в режиме RN при добавлении из AddMenu по умолчанию создаётся с `layout="grid"` (для RN удобнее флекс/сетка); в режиме Web — `layout="block"`.
     - **LayoutAccordion (вкладка «Стили»):** в режиме RN отображается только работа с **флексом** (одна кнопка «Флекс»), без выбора «Блок» / «Сетка» / «Абс. позиция» и без полей Columns/Rows.
   - Имена компонентов и структура JSON единые; мобильное приложение по этому же JSON рендерит экран нативно (см. п. 5).

5. **Мобильное приложение: два источника контента**  
   В мобилке действует правило:
   - **Контент из конструктора для RN** (`content_mobile`) — рендерится **нативно** (по JSON в RN‑компоненты).
   - **Контент из веб‑конструктора** (`content`) — показывается через **WebView** (загрузка URL веб‑сайта).
   Бэкенд в будущем реализует различие источников, и по **флагу** (в ответе API или в метаданных страницы) приложение будет понимать, какой контент отдавать — и выбирать режим отображения: нативный рендер по `content_mobile` или WebView по веб‑странице.

### 4. Стиль кода для `builder`

Чтобы `builder` и `site-runtime-ssr` выглядели и ощущались одинаково:

- **Функции**
  - Предпочитаем **стрелочные функции**:
    - компоненты вида `export const Text = (...) => { ... }`,
    - коллбеки (`map`, `filter`, обработчики событий).
  - Обычные `function` используем только при явной необходимости работы с `this`/контекстом.

- **Переиспользуемые enum и типы**
  - Выносить в файлы **builder.enum**: `src/pages/builder/builder.enum.ts` (режимы конструктора, например `MODE_TYPE`), `src/builder.enum.ts` (типы раскладки, `PlaceItemsValue`, `BlockLayoutMode`, `GridAutoFlow` и т.п.). Если добавляются новые enum или типы, которые используются в нескольких файлах билдера, добавлять их в соответствующий builder.enum.

- **Типы: `interface` vs `type`**
  - Для пропсов, контекстов и объектных структур — **`interface` по умолчанию**.
  - `type` используем для:
    - union/intersection/utility типов,
    - alias’ов, где `type` реально удобнее/короче.

- **Форматирование типов**
  - В интерфейсах и объектных `type` **после каждой строки ставим `;`**:
    - `field: string;`
    - `optional?: number;`
  - Это синхронизирует стиль с `site-runtime-ssr` и упрощает чтение/диффы.

- **React‑хуки**
  - Предпочитаем **импортировать хуки из `react` поимённо** и вызывать их напрямую:
    - `import { useState, useCallback, useEffect } from "react"`
    - `useState(...)`, `useCallback(...)` и т.п.
  - Не используем стиль `React.useState`, `React.useCallback` и т.д.,
    чтобы код был короче и единообразнее.

- **Точки с запятой**
  - Не используем «leading semicolon» (`;foo()` на новой строке).
  - Пишем по классике:
    - `const x = 1;`
    - `foo();`
  - Если нужно разорвать выражения — просто ставим `;` в конце строки,
    а следующую строку начинаем без ведущей точки с запятой.

- **Дополнительные правила для компонентов и MUI (общие)**
  - **Имена интерфейсов пропсов**
    - Если компонент живёт в отдельном файле и его интерфейс пропсов **используется только в этом файле** — называем интерфейс просто `Props`.
    - Если интерфейс переиспользуется в нескольких файлах — даём более предметное имя (`ImageSettingsProps`, `TextSettingsProps` и т.п.).
  - **Typography**
    - Не используем `variant` у `Typography`, предпочитаем задавать размер/линейку через `sx` (и при необходимости через токены `typography` из темы).
    - Пример: `sx={{ fontSize: "12px", lineHeight: "16px" }}` вместо `variant="body2"`.
  - **MUI `sx` и CSS‑свойства**
    - Не используем сокращения вроде `p`, `m`, `pt`, `pb` и т.п. в `sx`.
    - Всегда пишем полные имена свойств: `padding`, `marginTop`, `marginBottom` и т.д.
    - Не используем числовые «шкалы» MUI (`p: 1`, `m: 2`) — указываем значения явно в px.
    - Для всех свойств с размерами пишем строку с единицами: `height: "20px"`, `padding: "8px"`, `gap: "8px"`, а не просто `height: 20`.
  - **Комментарии**
    - Для описательных комментариев в коде (над функциями, хендлерами, блоками логики) — стиль `/** ... */`.
    - В JSX‑разметке — только `{/* ... */}`.
  - **MUI‑обёртки и переиспользуемые элементы**
    - В правой панели для аккордеонов настроек используем общие обёртки (`MUIAccordion`, `StyledAccordionDetails` и т.п.), а не «сырые» `Accordion`/`AccordionDetails` — это держит единый визуальный стиль.
    - Для иконок и управляющих элементов (например, переключатель Manual / Collection через `IconButton`) используем общие иконки из `core/components/Icons` и цвета из `COLORS`, а не инлайновые SVG/цвета.

### 6. Переиспользуемые UI‑паттерны и общие настройки компонентов

- **InlineSettingsModal**
  - Файл: `src/craft/InlineSettingsModal.tsx`.
  - Используется для всех «инлайн‑модалок» компонент‑настроек (сейчас: `Text`, `ContentList`, `Image`).
  - Пропсы:
    - `open`: boolean — отображать модалку или нет.
    - `title`: string — заголовок в шапке.
    - `top`, `left`: координаты (обычно берутся из `badgeRef.getBoundingClientRect()`).
    - `onClose`: колбэк закрытия.
    - `onShowAllSettings?`: опциональный колбэк для кнопки «Показать все настройки» внизу.
    - `children`: произвольный контент (формы, кнопки и т.п.).
  - Важные детали:
    - Вешает `mousedown`‑обработчик в **capture‑phase** на саму модалку, чтобы гасить события и не давать Craft.js/другим обработчикам перехватывать клики внутри.
    - Отдельным `useEffect` слушает клики по документу и закрывает модалку **только** при клике вне неё.
    - Стандартный layout: шапка (title + ✕), контент, снизу кнопка «Показать все настройки» (если передан `onShowAllSettings`).

- **InlineSettingsBadge**
  - Файл: `src/craft/InlineSettingsBadge.tsx`.
  - Переиспользуемая «бирка» над компонентом (иконка + название + шестерёнка).
  - Пропсы:
    - `icon?`: ReactNode — левая иконка (пример: `T`, `🖼`, `CL` или H1).
    - `label`: string — текст/название бирки.
    - `showSettingsButton?`: boolean — показывать ли шестерёнку (по умолчанию `true`).
    - `onSettingsClick?`: колбэк при клике на шестерёнку (обычно открывает `InlineSettingsModal`).
    - `maxWidth?`: максимальная ширина бирки (обрезка по `…`).
  - Важные детали:
    - Позиционируется абсолютно над компонентом (`top` отрицательный, чтобы «сидеть» на бордере).
    - Гасит `onMouseDown` на контейнере, чтобы клики по бирке не сбивали выделение/drag.
    - На кнопку‑шестерёнку вешает **native listeners в capture‑phase** (`mousedown` и `click`), чтобы гарантированно вызвать `onSettingsClick` до Craft.js и не дать ему перехватить события.
  - Общие договорённости по биркам:
    - **Body** — единственный компонент без бирки.
    - **Все остальные Craft‑компоненты**, которые можно перетащить на холст, при выборе показывают `InlineSettingsBadge`:
      - простые элементы без локальных настроек (например, `Block`, `Text` вне `ContentList`) — бирка только с типом и иконкой, **без** шестерёнки (`showSettingsButton = false`);
      - «сложные» элементы с настройками (сейчас: `Text` внутри `ContentList`, `Image`, `ContentList`, `LinkText`) — бирка **со шестерёнкой**, открывающей `InlineSettingsModal`.
    - Бирка должна появляться **по клику/выбору элемента** на холсте, а не постоянно.

- **settingsCraftComponents**
  - Папка: `src/pages/builder/settingsCraftComponents`.
  - Назначение:
    - Хранит **общие React‑компоненты с настройками** для Craft‑компонентов, которые
      одновременно показываются:
      - в инлайн‑модалках (`InlineSettingsModal`) прямо на холсте;
      - в правой панели, во вкладке **«Настройки»**.
    - Избегаем дублирования разметки/логики: один и тот же компонент рендерится в двух местах.
  - Пример:
    - `ImageSettingsFields.tsx`:
      - отображает:
        - URL изображения (Manual),
        - переключатель Manual / Collection,
        - выбор поля коллекции (если компонент внутри `ContentList`),
        - Width / Height (Auto / px).
      - использует `useEditor().actions.setProp` для изменения пропсов выбранного Craft‑узла,
        поэтому любые изменения сразу отражаются и на холсте, и в другом месте, где эти настройки
        отображаются.
      - умеет работать как:
        - простой блок (в модалке `Image`),
        - MUI‑аккордеон (в правой панели, вкладка «Настройки», если выбран `Image`).
      - для правой панели сам вычисляет, находится ли `Image` внутри `ContentList`:
        - поднимается по `ancestors(true)` через Craft‑`query`,
        - ищет предка с `displayName === "ContentList"` (через `resolveNodeDisplayName`),
        - берёт его `props.selectedSource` как `content_type_id`.
  - Договорённость:
    - Все новые настройки, которые должны быть **и в модалке, и в правом табе**, выносим в отдельный
      компонент в `settingsCraftComponents` и переиспользуем, вместо копирования JSX и логики.

- **craftSettingsControls**
  - Папка: `src/pages/builder/components/craftSettingsControls`.
  - Назначение:
    - Набор **атомарных UI‑контролов** для настроек (инпуты, селекты, группы кнопок, выбор цвета),
      которые переиспользуются в:
      - `settingsCraftComponents` (вкладка «Настройки», инлайн‑модалки),
      - `craftStylesComponents` (вкладка «Стили» правой панели),
      - при необходимости — в других частях конструктора.
    - Держат единый визуальный стиль для всех форм настроек (шрифты, отступы, цвета, стрелки и т.п.).
  - Компоненты:
    - `CraftSettingsInput`
      - Лейбл слева + текстовый/числовой инпут справа.
      - Используется для простых скалярных значений (например, `Size`, `Height`, `Stroke` width).
    - `CraftSettingsSelect`
      - Лейбл слева + `select` справа.
      - Кастомная стрелка на основе `ChevronRightIcon`, повернутая вниз и окрашенная в `COLORS.gray700`.
      - Применяется для выбора предопределённых вариантов (например, `Font`, `Weight`).
    - `CraftSettingsButtonGroup`
      - Лейбл слева + группа **взаимоисключающих** кнопок справа (один активный вариант).
      - Кнопки фиксированного размера, фон `purple100` / `white` в зависимости от выбранного состояния.
      - Используется для одноразовых переключателей вроде `Align`, `Capitalize`.
    - `CraftSettingsMultiToggleGroup`
      - Лейбл слева + группа кнопок, каждая из которых может быть включена/выключена независимо.
      - Применяется для множественного выбора формата текста (`S`, `U`, `I` для зачеркивания, подчёркивания, курсива).
    - `CraftSettingsColorField`
      - Лейбл слева + единый блок справа:
        - квадрат‑swatch текущего цвета,
        - поверх него скрытый `input type="color"`,
        - справа текстовое поле с hex‑значением (`#000000` и т.п.).
      - Используется для выбора цвета текста и обводки (`Color`, `Stroke` color) с поддержкой ручного ввода.
  - Договорённость:
    - Во всех новых настройках **не используем «сырые» `input`/`select`** в панелях; вместо этого
      берём соответствующий компонент из `craftSettingsControls`.
    - Если появляется новый повторяющийся паттерн (например, range‑инпут, toggle‑switch и т.п.),
      сначала выносим его в `craftSettingsControls`, а уже потом подключаем в `settingsCraftComponents` /
      `craftStylesComponents`.

- **craftStylesComponents**
  - Папка: `src/pages/builder/craftStylesComponents`.
  - Назначение:
    - Компоненты, которые описывают **стили для craft‑компонентов** и используются **только** во вкладке
      «Стили» правой панели.
    - Сейчас здесь лежат:
      - `LayoutAccordion` — управление layout (`block` / `flex` / `grid` / `absolute`).
        - В **режиме RN** отображается только вариант «Флекс» (без блок/сетка/абсолют и без полей Columns/Rows).
        - **Важно:** `display` и `position` — разные CSS‑свойства, но в одном блоке «Расположение»
          задаём оба. Сейчас `layout="absolute"` → `position: absolute` + `display: block`;
          остальные режимы задают только `display`. TODO: продумать, как лучше разделить или
          оформить настройки display vs position.
      - `SpacingAccordion` — визуальный контрол для `margin` / `padding`.
      - `BordersAccordion` — управление границами (стороны, ширина, цвет, opacity).
      - `TypographyAccordion` — типографика для текстовых компонентов (`Text`, `LinkText`).
  - Отличие от `settingsCraftComponents`:
    - `settingsCraftComponents` — «функциональные» настройки компонента (источники данных, привязки,
      бизнес‑логика), которые дублируются между инлайн‑модалкой и вкладкой «Настройки».
    - `craftStylesComponents` — **чисто визуальные стили**, которые живут только в табе «Стили» и не
      используются в модалках.

### 5. Как дальше развивать документацию

- При добавлении новых Craft‑компонентов:
  - описывать:
    - их назначение в билдере,
    - ожидаемое поведение в рентайме и мобилке,
    - особенности сериализации (что попадает в JSON, какие пропсы критичны).
- При изменении существующих:
  - фиксировать изменения контракта (новые пропсы, изменения структуры),
  - отмечать, как это влияет на `site-runtime-ssr` и на будущую мобильную реализацию.
