 Апи /content - даем кастомный контент, тот который заводит пользователь (новости, доп услуги и т.д.). Оснвной же идей внешних сайтов является продажа товаров. Товары приходят из другого апи. Нам нужно будет объединить у себя в проекте эти вещи. То есть например ContentList в списке коллекций для заполнения будет предлогать сущности из /content и /product

# Товары поставщика на внешних сайтах

Работа с товарами в рамках сайтов затрагивает две сущности: товары `Products` и контент для них - `ContentDistributorProducts`.

Товары поставщика существуют в рамках всей системы, а контент для товаров создаётся в рамках конкретного сайта. При этом для работы с товарами в рамках сайта необходимо использовать совместно следующие подсистемы:

- клиентское API v3 для b2c/b2b, для отображения товаров в клиентском интерфейсе: https://dev-api.cezyo.com/v3/docs
- API экстранета v2 для работы с `Products`: https://doc.cezyo.com/v2/#/products
- API сайтов v3 для работы с контентом в клиентском интерфейсе и в экстранете для конкретного сайта https://dev-api.cezyo.com/v3/sites/docs

# Структура данных

Сущность `Products` описана в других частях документации и реализация сайтов её не затрагивает.

Сущность `ContentDistributorProducts` структурно аналогична материалам типов контента на сайтах. Она всегда привязана к сайту через `site_id` и к товару поставщика через `product_id`. Комбинация `site_id + product_id` уникальна в рамках всей системы - для каждого сайта можно задать только один контент товара поставщика для конкретного товара.

К `ContentDistributorProducts` могут быть присоединены поля контента, аналогичные полям типа контента и имеющими ту же структуру. Основная разница - набор полей один на весь сайт (в отличие от набора полей на каждый тип контента).

# Логика работы клиента

Основной принцип работы таков: контент товара поставщика дополняет и переопределяет данные товара поставщика. Например, для конкретного сайта и конкретного товара поставщика, можно задать собственное название, slug, активность, даты активности и так далее. Та часть запросов, которая идёт на API b2b/b2c будет возвращать уже переопределённые данные в нужном виде. Однако всё то, что не вписывается в рамки текущего API b2b/b2c, нужно получать через API sites.

Например, вывод каталога товаров поставщика на клиенте предполагается таким:

1. Запрос на `/v3/client/catalog/products/full` для получения товаров поставщика с заголовком `Site-Id: SITE_ID`. В ответ придут только те товары, которые не деактивированы в рамках указанного сайта, а в качестве названий и `slug` будут использоваться данные их контента для сайта, если они есть. То есть, например, если есть два товара, и для одной создан `ContentDistributorProducts`, а для другого- нет, то придёт для первого товара всё то, что определено в созданном `ContentDistributorProducts`, а для второй - обычные данные.
2. Далее, имея на руках `ID` товара поставщика, которые надо вывести, создаётся запрос на `/v3/sites/{domain}/content/distributor/products/` с указанными ID в фильтре, и получаются дополнительные поля контента и прочая информация, которая задана в рамках сайта.
3. Эти данные необходимо скомбинировать и вывести в нужном виде.

Активность товаров на данный момент работает так: активным в рамках сайта считается товар, который активен на основном проекте Cezyo, у которого не проставлена `active=false` в `ContentDistributorProducts`, и у которого либо не выставлены `active_from` + `active_to`, либо текущая дата попадает в этот диапазон.

В рамках категории поставщика на сайте можно задать собственную сортировку товаров в ней. Для того, чтобы товары в каталоге отдавались в нужном порядке, необходимо к методам каталога передавать не обычный `sort`, а query-параметр `site_sort_category_id` равный ID данной категории. При правильной комбинации заголовка `Site-Id` и параметра `site_sort_category_id`, товары будут отсортированы, если возможно, в выставленном в экстранете порядке. При этом надо учитывать, что API клиента по-умолчанию работает со всеми поставщиками сразу, и не забывать про `filter` `distributor_id`

Следующие методы в API v3 переопределяют контент через Site-Id:

- `/v3/client/catalog/products/full` Кроме того, этот метод обрабатывает `site_sort_category_id` для сортировки товаров
- `/v3/client/products/`
- `/v3/client/products/{id}`
- `/v3/client/products/{id}/full`

При этом все остальные методы каталога, например, работающие со статистикой фильтров, категориями, брендами и так далее, тоже учитывают `Site-Id,` если возможно. Например, методы для вывода доступных значений фильтра должны учитывать параметры активности, если указан нужный Site-id, так что желательно его использовать везде.
