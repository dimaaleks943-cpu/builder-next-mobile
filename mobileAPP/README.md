## Мобильное приложение (Expo + React Native)

Этот каталог зарезервирован под мобильное приложение, которое будет клиентом к нашему конструктору и Next.js‑витрине.

Цель: отображать на телефоне те же страницы и блоки, которые собираются в `builder` и рендерятся в веб‑витрине, используя один и тот же JSON c бэкенда.

Целевые платформы: **Android и iOS** (одна кодовая база на React Native/Expo).

### Архитектурные цели первой версии

- **Клиент к существующему API**  
  - Подключиться к тому же backend‑API, которое использует конструктор/Next.js.  
  - Получать JSON‑описание страниц и контента и рендерить его в виде RN‑экранов.

- **Минимальный стек:**  
  - Expo (managed workflow) + React Native + TypeScript.  
  - React Navigation для экранов/стеков.  
  - HTTP‑клиент (`fetch` или `axios`) с вынесенной конфигурацией base URL и токенов.

- **Совместимость с конструктором:**  
  - Не дублировать бизнес‑логику: максимально переиспользовать контракты (JSON‑структуры) и соглашения, уже описанные в `builder/DEV_NOTES.md` и рентайме.  
  - Отрисовывать те же блоки/структуры (Body/Block/Text/ContentList/и т.п.) в мобильном UI, но с адаптированными компонентами RN.

### План работ (высокоуровневый)

1. **Инициализация проекта Expo**
   - Создать новый Expo‑проект в каталоге `mobileApp/` (TypeScript шаблон).
   - Настроить базовую структуру: `src/screens`, `src/components`, `src/api`, `src/navigation`.
   - Обеспечить возможность запуска под **Android** (эмулятор/устройство через Android Studio) и под **iOS** (через Xcode/симулятор на macOS).

2. **Настройка API‑слоя**
   - Вынести base URL и ключевые параметры в конфиг/переменные окружения Expo.
   - Реализовать простой клиент и первый запрос к тому же API, что использует `site-runtime-ssr` (список страниц сайта и страница по slug).

3. **Базовый рендер страницы по JSON**
   - Подключить навигацию (stack).  
   - Сделать экран, который:
     - тянет JSON страницы по ID,  
     - отображает его в виде примитивного layout’а (пока без полного соответствия web‑рендеру).

4. **Документация**
   - Обновить этот `README` после инициализации (конкретные команды `npm/yarn`, требования по Node/Expo‑CLI и т.п.).  
   - Зафиксировать минимальный контракт JSON, который мобильное приложение ожидает от API (ссылкой на существующую документацию рентайма).

### Статус

- Expo‑проект инициализирован (Expo + React Native + TypeScript).
- В `src/api/config.ts` и `src/api/runtimeSite.ts` настроен базовый API‑слой для работы с runtime‑API (тем же, что использует `site-runtime-ssr`).
  - Ожидаемые переменные окружения (в Expo достаточно создать `.env` в корне `mobileApp`):

    ```env
    EXPO_PUBLIC_RUNTIME_API_URL=https://dev-api.cezyo.com
    EXPO_PUBLIC_SITE_DOMAIN=marketflow.store
    EXPO_PUBLIC_SITE_WEB_URL=https://marketflow.store
    ```

  - `EXPO_PUBLIC_SITE_WEB_URL` — базовый URL веб‑сайта (site-runtime-ssr) для отображения страниц в WebView. По умолчанию: `https://{EXPO_PUBLIC_SITE_DOMAIN}`.

  - **Локальная разработка (WebView):** Next.js запущен локально (например `http://marketflow.store:3000` с записью в hosts `127.0.0.1 marketflow.store`). В **Android-эмуляторе** твой `hosts` не используется, поэтому в `.env` укажи адрес хоста и порт: `EXPO_PUBLIC_SITE_WEB_URL=http://10.0.2.2:3000` — `10.0.2.2` в эмуляторе это твой компьютер (127.0.0.1). Тогда в приложении откроются те же страницы, что и в браузере по `http://marketflow.store:3000/...`. В **iOS-симуляторе** обычно подходит `http://localhost:3000`. Домен `marketflow.store` в DNS может не существовать — тогда без IP/10.0.2.2 будет ошибка `ERR_NAME_NOT_RESOLVED`.

  - **Сайт открыт только через hosts на ПК:** эмулятор не использует твой `hosts` (у него свой DNS). Чтобы WebView открывал тот же сервер, задай в `.env` URL по **IP**: `EXPO_PUBLIC_SITE_WEB_URL=http://168.119.15.58` (или `https://...`, если сервер отдаёт сайт по IP; при HTTPS возможна ошибка сертификата, т.к. сертификат выдан на домен, а не на IP).

  - Пример использования в коде:

    ```ts
    import { fetchSitePageBySlug } from "./src/api/runtimeSite";

    const page = await fetchSitePageBySlug("/"); // корневая страница
    ```

Текущий режим: **все страницы отображаются в WebView** (загрузка URL веб‑сайта). В дальнейшем бэкенд добавит флаг, по которому будет выбираться режим: WebView или нативный RN‑рендер.
